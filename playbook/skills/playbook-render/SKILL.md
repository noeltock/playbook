# playbook-render

Pure-function regeneration of all playbook view files from their JSONL sources. Reads `decisions.jsonl` and `errors.jsonl`, resolves the last-record-wins JSONL pattern, and writes `DECISIONS.md` and `ERRORS.md`.

## When to use

- Called automatically by `/playbook-decide` and `/playbook-error` after writing new records.
- Called directly by the user to force-sync if view files are stale or corrupted.
- Called by `/playbook-sync` after absorbing drift.

Do not modify `decisions.jsonl` or `errors.jsonl` directly — this skill is read-only with respect to the source files.

## Steps

### 1. Resolve decisions

Read `decisions.jsonl` line by line. For each line, parse the JSON and index by `id`. **The last record for a given `id` wins** (this is the JSONL append-only update pattern). The result is a map of `id → record`.

If the file is absent or empty, treat as an empty set.

### 2. Render DECISIONS.md

Build `DECISIONS.md` with this structure:

```markdown
<!-- auto-generated by playbook-render — do not edit by hand -->
# Decisions

This file is auto-generated from `decisions.jsonl`. Edit records there, then run `/playbook-render`.

## Summary

| ID | Title | Status | Date |
|---|---|---|---|
| DEC-001 | Use PostgreSQL for primary storage | accepted | 2026-05-19 |
| DEC-002 | Adopt Tailwind CSS | superseded | 2026-04-10 |

## Records

### DEC-001: Use PostgreSQL for primary storage

**Status:** accepted
**Date:** 2026-05-19

#### Context

...

#### Decision

...

#### Consequences

...

---
```

Ordering rules:
- `accepted` decisions first, sorted by `id` ascending.
- `superseded` decisions at the bottom.
- Omit decisions with status `"draft"` from the summary table but include them in records with a `[DRAFT]` prefix.

### 3. Resolve errors

Read `errors.jsonl` line by line. Same last-record-wins logic indexed by `id`.

### 4. Render ERRORS.md

Build `ERRORS.md` with this structure:

```markdown
<!-- auto-generated by playbook-render — do not edit by hand -->
# Errors & Warnings

This file is auto-generated from `errors.jsonl`. To record a new error, use `/playbook-error`.

## Unresolved & Recurrent

> These require attention.

### ERR-001: npm ci fails when node_modules is a symlink

**Kind:** error | **Resolution:** unresolved
**Recorded:** 2026-05-19

**Context:** ...

**Remedy:** unknown

---

## Workarounds

...

## Resolved

...
```

Grouping and ordering:
1. `resolution: "unresolved"` and `resolution: "recurrent"` — top section, sorted by `ts` descending (newest first).
2. `resolution: "workaround"` — middle section.
3. `resolved: true` and `resolution: "permanent"` — bottom section, collapsed / de-emphasised.

### 5. Write output files

Write `DECISIONS.md` and `ERRORS.md` to the project root (or the output paths configured in `playbook/playbook.yaml` under `outputs`). Overwrite existing files.

### 6. Write viewer content (if applicable)

If `playbook/viewer/src/content/` exists, write the same content there for the Fumadocs build:
- `playbook/viewer/src/content/decisions.mdx`
- `playbook/viewer/src/content/errors.mdx`

The MDX files are the same content as the `.md` files with a frontmatter block prepended:

```mdx
---
title: Decisions
description: Architecture and design decisions recorded by the team.
---
```

### 7. Trigger viewer rebuild

Run `${PLAYBOOK_ROOT}/hooks/rebuild-viewer.sh`. The script handles the logic of whether a build is actually needed (dev server running, output tracked by git, etc.).
