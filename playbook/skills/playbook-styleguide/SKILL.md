# playbook-styleguide

Create `STYLEGUIDE.md` and `tokens.json` from scratch by scanning the codebase, or maintain them on subsequent runs by detecting near-duplicates and contradictions between the files and the codebase.

## When to use

- User asks to create or update the styleguide.
- Called by `/playbook-sync` when new visual primitives are detected (new CSS custom properties, Tailwind config changes, new component files).
- Before a design review or handoff, to ensure the styleguide reflects the current codebase.

---

## Steps — create (STYLEGUIDE.md or tokens.json absent)

### 1. Scan for visual primitives

Search the codebase for:

**CSS custom properties**
```css
:root {
  --color-primary: #3b82f6;
  --font-sans: 'Inter', sans-serif;
  --radius-md: 6px;
}
```
Look in `*.css`, `*.scss`, `*.sass`, `global.css`, `styles/`, `app/globals.css`.

**Tailwind config**
Read `tailwind.config.js`, `tailwind.config.ts`, or `tailwind.config.mjs`. Extract:
- `theme.colors` / `theme.extend.colors`
- `theme.fontFamily` / `theme.extend.fontFamily`
- `theme.spacing` / `theme.extend.spacing`
- `theme.borderRadius` / `theme.extend.borderRadius`
- `theme.fontSize` / `theme.extend.fontSize`

**Design token files**
Look for `tokens.json`, `tokens/*.json`, `design-tokens.json`, or files with `$value` / `$type` keys (Token Studio format).

**Component patterns**
Scan `components/`, `src/components/`, `ui/` for common patterns:
- Button variants and sizes
- Card, Badge, Input, Alert components
- Extract prop types or class variants if using CVA, Radix, or similar.

### 2. Draft STYLEGUIDE.md

Structure the draft with these sections:

```markdown
# Styleguide

> Auto-generated by /playbook-styleguide. Last updated: YYYY-MM-DD.

## Colours

| Token | Value | Usage |
|---|---|---|
| `--color-primary` | `#3b82f6` | CTAs, links, focus rings |

## Typography

| Token | Value |
|---|---|
| `--font-sans` | `'Inter', sans-serif` |
| `--font-size-base` | `16px` |

## Spacing

...

## Border Radius

...

## Components

### Button

Variants: `default`, `outline`, `ghost`, `destructive`
Sizes: `sm`, `md`, `lg`

...

## Signature elements

Distinctive visual patterns that define the product's identity (e.g., a specific gradient, an icon style, a custom animation).
```

Fill in what was found in step 1. Leave a `<!-- TODO -->` comment for sections with no data found.

### 3. Draft tokens.json

Output a DTCG-compliant token file:

```json
{
  "color": {
    "primary": {
      "$value": "#3b82f6",
      "$type": "color",
      "$description": "Primary brand colour. Used for CTAs and focus rings."
    }
  },
  "font": {
    "family": {
      "sans": {
        "$value": "'Inter', sans-serif",
        "$type": "fontFamily"
      }
    }
  }
}
```

Map all discovered primitives to DTCG groups: `color`, `font`, `spacing`, `radius`, `shadow`, `typography`.

### 4. Confirm before writing

Show the user:
- A summary of what was found ("Found 12 colour tokens, 3 font families, 8 spacing values, 4 components").
- The drafted `STYLEGUIDE.md` and `tokens.json`.

Ask: "Write these files?" Proceed only on confirmation.

### 5. Write both files

Write `STYLEGUIDE.md` to the project root and `tokens.json` to the project root (or `playbook/tokens.json` if a `playbook/` directory is the convention). Match the project's existing file placement conventions.

---

## Steps — maintain / detect conflicts

### 1. Compare tokens.json against STYLEGUIDE.md

For each token in `tokens.json`, check if its value matches the value shown in `STYLEGUIDE.md`. Flag any mismatches as potential contradictions.

### 2. Scan codebase for near-duplicates

Look for values that are suspiciously similar:

- **Colours**: two hex values within a small perceptual distance (e.g., `#3b82f6` vs `#3b81f5` — likely a typo or unintentional fork).
- **Border radius**: two values within 2px of each other (e.g., `6px` and `7px` both used on buttons).
- **Font sizes**: two values within 1px of each other (e.g., `14px` and `15px` used for body text).
- **Spacing**: near-duplicate spacing values used for the same component type.

Use grep or AST scan to locate usages of the near-duplicate values.

### 3. Present findings

For each finding, show:

```
Found two similar border-radius values: 6px (--radius-sm) and 8px (--radius-md) — both used on card components.
Options:
  1. Keep both (intentional distinction)
  2. Merge to 6px
  3. Merge to 8px
  4. Rename to clarify purpose
```

Wait for the user's choice before proceeding.

### 4. Apply choices

- Update `tokens.json` with the resolved values.
- Update `STYLEGUIDE.md` to reflect any renames or merges.
- If the user chose to merge, find and update usages in the codebase (only if they explicitly say so — do not touch codebase without confirmation).

### 5. Trigger viewer rebuild

Run `${PLAYBOOK_ROOT}/hooks/rebuild-viewer.sh`.
